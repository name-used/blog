SpringCloud学习笔记

本文概述部分为个人理解,欢迎指正 (反正评论不能用)

微服务是什么

  • 是一种软件架构
  • 是一族独立运行的小项目
  • 是单一应用程序的拆分
  • 满足以下关键词:
    • 微型服务
    • 独立进程
    • 轻量交互
    • 独立部署 (强调服务的启用不必依赖其它服务)

为什么微服务

  • 一个项目只做一件事
    • 高内聚,松耦合
    • 聚焦业务逻辑
    • 减少代码量
      • Bug = (Code)^2
      • 小团队即可开发,降低管理成本
  • 一个项目是具有功能意义的独立的服务
    • 不要求开发语言一致
    • 某个项目触雷不影响其它项目的正常运行
    • 易于集成第三方实现自动部署
    • 允许更灵活的开发,如使用新技术

微服务的缺点

  • 分布式系统复杂性
    • 系统部署依赖
    • 系统间通讯
    • 数据一致性
  • 服务过多增加运维压力
    • 系统集成测试
    • 性能监控

微服务怎么做

  • 服务治理
    • SpringCloud
    • Dubbo
  • 服务注册
    • Eureka -> (高可用性,分区容错性)
    • Zookeeper -> (强一致性,分区容错性)
  • 服务调用
    • restful-api
    • remote-procedure-call
  • 服务负载均衡 ((Load-Balance))
    • 客户端负载均衡
      • Ribbon
      • Feign
    • 服务器端负载均衡
      • Nginx
      • F5
  • 服务监控
    • dashboard
  • ~…(To Be Continue)…~

SpringCloud是什么

  • 是微服务框架的一种落地实现
  • 是微服务所需技术的一种集合体

为什么选SpringCloud

  • 框架成熟度
  • 社区热度
  • 可维护性
  • 解决方案
  • 学习曲线

SpringCloud能做什么

  • 分布式版本控制配置
  • 服务注册与发现
  • 路由
  • 服务到服务的调用
  • 负载均衡配置
  • 断路器
  • 分布式消息管理
  • ~…(To Be Continue)…~

      • 在Maven中引入相关依赖

      • 在yml中配置相关属性
      • 修改业务逻辑
    • 哪些组件

      • 负载均衡: Ribbon/Feign
      • 断路器: Hystrix
      • 监控: dashboard
      • 路由: Zuul
      • 分布式配置中心: SpringCloud-Config
  • 启用Eureka
    • Eureka是一个独立的服务,因此需要新建它
    • pom
      • Eureka-Server
        • org.springframework.cloud
        • spring-cloud-starter-netflix-eureka-server
      • 不确定的依赖项
        • hystrix
        • com.sun.xml.bind::jaxb-impl::2.1.2
      • 热部署仅需devtools,不再需要springloaded
    • application.yml
      • server.port
      • eureka.instance.hostname ((我不明白它的作用))
      • eureka.client
        • register-with-eureka: false
        • fetch-registry: false
        • service-url.defaultZone: http://${eureka.instance.hostname}:${server.port}/eureka/ ((我不明白它的作用))
    • 启动类
      • @EnableEurekaServer
    • 配置完成,eureka启动正常
    • 将Provider注册到Eureka中
      • pom
        • spring-cloud-starter-netflix-eureka-client
      • application.yml
        • eureka.client.service-url.defaultZone: http://url_to_eureka
        • eureka.instance.instance-id: 修改此名称以复写eureka服务注册界面上的url显示
        • eureka.instance.prefer-ip-address: 使用ipAddress而非hostname
      • 启动类
        • 可以不添加
        • 可以添加 @EnableEurekaClient
        • 也可以添加 @EnableDiscoveryClient
      • 非必须:关于服务发现,可以在Controller中添加路径进行测试,是一种自身信息的对外展现
        • private DiscoveryClient client;
        • @GetMapping(“/app_name/discovery”)
        • // do anything
        • return client;
      • 启用actuator
        • actuator负责服务的信息发现,因此修改Provider项目
        • pom添加spring-boot-starter-actuator
        • application.yml添加info
        • 在pom的build中添加插件和命令
          • resources->resource->
            • directory->src/main/resources
            • filtering->true
          • plugins->plugin
            • groupId->org.apache.maven.plugins
            • artifactId->maven-resources-plugin
              -configuration->delimiters->delimiter->$
        • info使用上述定义的规则(如$project.version$)动态插值
    • 配置Eureka集群
      • 修改application.yml
        • 修改 eureka.instance.hostname 进行直观区分
          • 学习过程中可以直接修改hosts文件来完成这件事
          • 更进一步的,可以做一个端口映射,比如我认为nginx很不错
        • 修改 server.port 进行实质区分
          • 如果有多台机器,或者不同的虚拟机实现集群,则无需修改
        • 修改 eureka.client.service-url.defaultZone 使包含集群中的其它Eureka服务
      • 修改provider的application.yml
        • 修改 eureka.client.service-url.defaultZone 使包含每个Eureka服务
      • 启动后发现每个服务下都挂载了另外两个服务,以及都包含provider,则说明配置成功
  • 启用Ribbon
    • Ribbon是客户端负载均衡算法,因此修改Consumer项目
    • pom
      • Eureka-Client
        • spring-cloud-starter-netflix-eureka-client
      • Ribbon
        • spring-cloud-starter-netflix-ribbon
    • application.yml
      • eureka.client.register-with-eureka: false # 因为消费者不提供服务
      • eureka.client.service-url.defaultZone: [每个Eureka服务]
    • 启动类
      • @EnableEurekaClient
    • Controller
      • RestTemplate访问服务者时不再采用ip:port,而是采用服务名
    • 为RestTemplate的服务名解析提供支持,增加配置类ConfigBean
      • @Bean
      • @LoadBalanced
      • public RestTemplate getXXX(){return new RestTemplate();}
    • 配置Provider集群测试Ribbon是否生效
    • 自定义Ribbon的均衡算法,修改ConfigBean
      • @Bean
      • public IRule getXXX(){return new 算法的实现();}
        • 此处有一个坑,即,这个算法的实现必须是一个public class
        • 虽然并不清楚原理,但是显然Spring会尝试newInstance它
        • 所以使用匿名对象,非public类的实例,会导致失败
      • 公开课中提到:
        • 使用自定义算法需要单独使用一个配置类
        • 且这个配置类不能被@ComponentScan扫描到
        • 但这个问题的出现是因为”特殊化定制”
        • 即@RibbonClient(configuration=MyConfig.class)
        • 无必要的情况下,不写configuration即可
  • 启用Feign
    • Feign集成了Ribbon,因此拷贝一份Ribbon工程并修改
    • pom
      • Feign
        • spring-cloud-starter-openfeign
    • 启动类
      • @EnableFeignClients
    • 修改API工程 ((因为Feign是面向接口调用))
      • pom添加Feign
      • 接口添加注解 @FeignClient(“服务名”)
      • 接口添加方法(参照Provider的Controller)
    • 修改Controller
      • 去掉RestTemplate
      • 添加自定义的Feign接口
      • 重新实现业务逻辑
  • 启用Hystrix
    • Hystrix负责解决服务级联故障
      • 服务熔断: 服务者可以处理自身的故障(可能由扇出链路引起)
      • 服务降级: 消费者可以处理服务者故障
      • 与语法上进行异常处理的区别:
        • Hystrix会监控调用失败的情况,并决定是否熔断和降级
        • 服务熔断后,服务者接收访问时会快返回失败信息,不再执行自身方法
        • 服务降级后,消费者准备访问时会快速返回失败信息,不再进行访问请求
    • 服务熔断
      • 针对Provider使用,拷贝并修改Provider工程
      • 在Controller中使用@HystrixCommand(fallback=”函数名”)
      • 同一个Controller下按照上述指定的函数名写一个同参同返回值的方法
      • 鉴于上述操作的复杂性和不靠谱性质,除非研究源码尝试封装它
      • 否则仅考虑在个别方法上使用即可
    • 服务降级
      • 针对Consumer使用,修改API工程
      • 修改@FeignClient注解,添加fallbackFactory=自定义Fallback处理类.class
      • 新建Fallback处理类实现FallbackFactory<自定义接口>接口
      • Fallback处理类添加@Component注解
      • 重写create方法并在返回值中创建自定义接口的fallback实现
  • 启用HystrixDashboard
    • HystrixDashboard是独立的监控服务,因此新建项目
    • pom
      • hystrix-dashboard
        • spring-cloud-starter-netflix-hystrix-dashboard
      • hystrix
        • spring-cloud-starter-netflix-hystrix
        • 监控provider工程的/actuator/hystrix.stream需要此依赖
    • 启动类
      • @EnableHystrixDashboard
    • 测试和使用
      • 在监控服务的页面上搜索想要监控的服务ip:port/actuator/hystrix.stream
      • 使被监控的服务上由hystrix管理的方法得到访问
      • 刷新监控界面获得图形信息
  • 启用zuul
    • Zuul是独立的路由服务,因此新建项目
    • pom
      • zuul
        • spring-cloud-starter-netflix-zuul
      • eureka
        • spring-cloud-starter-netflix-eureka-client
        • 路由需要注册到eureka中使用,因此依赖eureka客户端
    • application.yml
      • zuul.routes.mydept.serviceId: 进行路由的服务名
      • zuul.routes.mydept.path: 进行路由的地址
      • zuul.ignored-services: 忽略的服务名(阻止通过服务名进行访问)
    • 启动类
      • @EnableZuulProxy
    • 测试和使用
      • …
  • 启用SpringCloud-Config
    • SpringCloud-Config是分布式配置中心,是一整套配置文件的解决方案
    • 新建配置中心,例如使用git仓库,创建和提交配置文件信息,格式如下
      • spring.profiles.active: [dev]
      • spring.profiles: dev &&正常配置
      • spring.profiles: test && 正常配置
    • 新建项目config-server
      • pom
        • spring-cloud-config-server
      • application.yml
        • spring.cloud.config.server.git.uri: 指定git仓库
      • 启动类
        • @EnableConfigServer
      • 测试访问
        • 依据官网格式要求,有数种不同的请求格式,因为yml格式最适合阅读,采用这种测试
        • http://config-server/application-dev.yml
        • http://config-server/application-test.yml
        • 其中application即配置文件的名字,最后一个-dev或者-test则是指定的profiles
        • 文件名中包含短杠并不影响访问
    • 新建项目config-client
      • 此项目用于测试配置是否能够被正常读取
      • pom
        • spring-cloud-config-client
        • spring-boot-starter-web
      • application.yml
        • 不用写东西,或仅写一个spring.application.name
      • bootstrap.yml
        • 关于
          • application是用户级资源配置项
          • bootstrap是系统级资源配置项,加载优先级更高
          • bootstrap属性有高优先级,默认情况下不会被本地配置覆盖
          • 既然application.yml的内容需要从远端获取,本地就使用bootstrap来保证配置分离
        • spring.cloud.config.name: 指定github上的资源名称(不含后缀)
        • spring.cloud.config.profile
        • spring.cloud.config.label: master
        • spring.cloud.config.uri: 指定config-server的服务地址
      • Controller
        • 使用@Value(“${配置项}”)注入属性
        • 写一个接口用于测试
    • 参照config-client修改拷贝的其它项目
    • 测试访问
